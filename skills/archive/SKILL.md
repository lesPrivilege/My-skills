# 歸檔工作流

用戶貼了一段內容。先判斷輸入類型，再走對應分支。

## 0. 讀取配置

讀取 `~/Scripts/archive-session/config.json`。分類對應：
- `技術與系統` → `tech.md`
- `組織、資本與權力` → `power.md`
- `認知、哲學與心智` → `mind.md`
- `數學、學習與方法論` → `method.md`
- `生活、決策與心智工具` → `life.md`
- `閱讀` → `reading.md`
- `項目` → `projects.md`

筆記寫入 `{vault}/archive/{short}.md`。記憶寫入 `config.memory_path`（Auto Memory 目錄，目前為 `~/.claude/projects/-Users-lesprivilege/memory/`）。

**重要區別**：前 5 類使用 `- **標題**：正文` 原子筆記格式；`reading.md` 使用自由格式（可含表格、序列、段落敘述），按日期分節追加，不要求每條原子化；`projects.md` 使用項目敘事格式（元數據 + 當前狀態 + 敘事 + 決策 + 洞察萃取），一個項目一個節。

**記憶更新機制**：直接讀寫 `config.memory_path` 目錄下的文件。讀取時掃描 `MEMORY.md` 索引和各 `.md` 文件；寫入時創建或更新對應 `.md` 文件並更新 `MEMORY.md` 索引。此機制對 Claude Code 和 OpenCode 均適用。注意：不動 `~/CLAUDE.md`（那是用戶手寫的規則文件）。

## 輸入判斷

優先級從高到低：

1. 檢查用戶貼文是否包含 **「Memory 操作：」** 行：
   - **包含** → 視為已結構化歸檔輸出，走正常流程（跳過壓縮步驟）

2. 檢查是否為 **項目敘事**（判據見下）：
   - **命中** → 走項目歸檔分支（Section C）
   - **不命中** → 視為原始長文本，先壓縮為結構化格式，經用戶確認後再繼續

### 項目敘事判據

滿足以下任意兩項即視為項目敘事：
- 內容呈**時間弧線**：發現 → 探索 → 迭代 → 閉合
- 包含**設計決策**（「決定 X 而非 Y，因為…」）
- 包含**問題-修復**循環（bug → 定位 → 修復 → 驗證）
- 包含**方案演進**（v1 → v2 → 最終方案）
- 包含**工具鏈搭建**過程（安裝 → 配置 → 調試 → 運行）

## A. 原始文本壓縮

用 `~/Archived.md` 的格式提示詞為模板，對貼文進行壓縮：

1. 分析貼文內容，提取：
   - 值得沉澱到 memory 的穩定事實、持久偏好、確認的目標、重要約束
   - 符合七類判據的原子化洞察（每條獨立、可脫離上下文理解）
   - **閱讀相關內容**（見第 6 類判據）：書目、作者分析、母題追蹤、跨文本對照、讀書方法等。閱讀類的提取導向可以更寬——不要求原子化，允許敘事序列、表格、重點摘錄
2. 按 `~/Archived.md` 的體例輸出結構化文本（Memory 操作摘要 + 六類筆記，含閱讀類）
3. 將結果展示給用戶，**等待用戶確認後再繼續**。用戶可以：
   - 直接確認
   - 要求修改（刪/改/增條目）
   - 放棄歸檔

**記憶原則**：壓縮分支**默認不動 memory**。如壓縮過程中識別到值得記憶的穩定事實，在輸出中以「建議記憶」標註，等用戶確認後再寫入。必須堅持最小 memory 原則——不記錄臨時性內容。

## B. 正常歸檔流程（用戶確認後 / 已結構化輸入）

### 1. 更新記憶

按用戶貼文中的 Memory 操作摘要，讀寫 `config.memory_path` 目錄：
- 讀取 `MEMORY.md` 索引，掃描已存在的記憶文件
- 新增條目 → 創建新 `.md` 文件，追加到 `MEMORY.md` 索引
- 修改條目 → 找到對應 `.md` 文件替換內容
- 刪除條目 → 刪除對應 `.md` 文件，從 `MEMORY.md` 索引移除

### 2. 解析筆記

在貼文中找到各分類下的筆記。每條格式：`- **標題**：正文`

### 3. 歸檔筆記（逐分類追加）

對前 5 類（tech / power / mind / method / life）：
   每條格式：`- **標題**：正文`

   **去重**（使用 hash 索引，避免讀完整個 .md 文件）：
   1. 將所有新筆記的標題收集起來，傳遞給 `idx.py dedup` 腳本進行批量去重：
      ```bash
      python3 ~/Scripts/archive-session/idx.py dedup <category> "標題1" "標題2" ...
      ```
      腳本輸出**不在索引中**的標題（即真正需要歸檔的新條目），每行一個。
   2. 對比輸入標題總數與腳本輸出的新標題數，差值即為跳過的重複項。

   **寫入**：
   3. 去重後的新條目，在 `{short}.md` 檔案末尾追加：
      ```markdown

      ## 歸檔 · {YYYY-MM-DD}
      - **標題**：正文

      - **標題**：正文
      ```
   4. 如檔案還不存在，先寫標題行（`# {分類名}`）再追加
   5. **寫入完成後，重建索引**：
      ```bash
      python3 ~/Scripts/archive-session/idx.py rebuild <category>
      ```

對第 6 類（reading）：
   格式不拘（段落、表格、序列、摘錄均可），按日期分節追加。
   1. 讀取 `archive/reading.md`（不存在則跳過）
   2. 檢查是否已存在 `## 歸檔 · {YYYY-MM-DD}` 標題行
   3. 不存在 → 在檔案末尾追加完整閱讀內容，以 `## 歸檔 · {YYYY-MM-DD}` 為節頭
   4. 已存在 → 追加續節：`## 歸檔 · {YYYY-MM-DD}（續）`，然後寫入新內容
   5. 如檔案還不存在，先寫 `# 閱讀回顧` 標題再追加

### 4. 交叉參照（輕量 · 純建議不寫入）

新條目歸檔完成後，在彙報前做一次輕量交叉參照：

1. **收集**本次新寫入的所有條目標題及對應分類（去重後的結果）
2. **讀取**所有已有分類的 `.idx` 文件（每個文件的 `hash|title` 行即完整條目列表）
3. **比對**：LLM 判斷哪些現有條目與新條目**可能相關**，判斷維度包括：
   - 同一概念的不同面向
   - 方法與案例
   - 補充或反駁（新條目挑戰或修正舊觀點）
   - 前因後果（新技術取代舊方案）
4. **不寫入**任何東西——結果僅作為彙報的附加資訊
5. 成本控制：如所有 `.idx` 行數總和 > 500 行，跳過此步驟

### 5. 彙報

簡短彙報（包含交叉參照建議）：
- 記憶更新摘要（增/改/刪）
- 各分類歸檔了多少新筆記，跳過了多少重複
- 交叉參照（如有匹配）：`「新條目 X」→ 可能相關：「現有條目 Y」(tech)、「現有條目 Z」(mind)`

## C. 項目歸檔流程

當輸入被判定為項目敘事時，走此分支。

**記憶原則**：項目歸檔**默認不動 memory**。如歸檔過程中識別到值得記憶的穩定事實（如「建置了本地 arxiv-browse skill」），在壓縮結果中以「建議記憶」標註，等用戶確認後再寫入。必須堅持最小 memory 原則——不記錄臨時性內容。

### 1. 壓縮為項目格式

用 `~/Archived.md` 的項目歸檔格式為模板，對貼文進行壓縮：

1. 提取項目元數據：名稱、週期、狀態、工具
2. 壓縮敘事為 5-10 段（每個轉折點一段，每段 1-3 句）
3. 提取關鍵決策（表格格式）
4. 識別「當前狀態」：已完成 / 進行中 / 待處理
5. 從敘事中萃取原子洞察，按前 6 類判據分類，標註應寫入的目標文件

### 2. 展示確認

將壓縮結果展示給用戶，**等待用戶確認後再繼續**。用戶可以：
- 直接確認
- 要求修改（調整敘事粒度、增刪決策、修改狀態）
- 放棄歸檔

### 3. 寫入 projects.md

用戶確認後：
1. 讀取 `archive/projects.md`
2. 檢查是否已存在同名項目（按 `## 歸檔 · {日期} · {項目名稱}` 匹配）
   - 已存在 → 更新「當前狀態」節，追加新敘事段和決策
   - 不存在 → 在檔案末尾追加完整項目節
3. 如檔案還不存在，先寫 `# 項目回顧` 標題再追加

### 4. 萃取原子洞察

從項目敘事中浮現的原子洞察，按現有六類寫入對應文件：
- 按各類的去重規則（標題匹配）避免重複
- 在 `projects.md` 的「方法論沉澱」節中標註每條去向：`→ tech.md（已歸檔）`
- 如某類無新增洞察則跳過

### 5. 輸出續行區塊

歸檔完成後，輸出續行區塊供用戶複製（不存儲）：

```
[項目續行] {項目名稱}

請先 review 此項目的「當前狀態」節，然後等我貼入補充上下文後繼續工作。

**焦點**：{用戶指定或 AI 建議的本次聚焦點}
```

### 6. 彙報

簡短彙報：
- 項目歸檔摘要（新項目 / 更新已有項目）
- 萃取了多少原子洞察，寫入了哪些分類文件
- 續行區塊（供複製）

## 設計原則

- **聲明式**：你只貼內容，不需要操作步驟
- **冪等**：同一條筆記貼兩次不重複寫入；同一項目歸檔更新而非重複
- **單檔瀏覽**：一分類一檔案，日期分節，閱讀流暢
- **可擴展**：config.json 獨立，後續加 RAG 只需加一個模塊
- **三模態**：已結構化輸出一鍵歸檔、原始文本先壓縮確認再歸檔、項目敘事走獨立分支
- **雙用途**：項目歸檔既是存儲記錄，也是跨 session 續行的上下文載體

---

## 格式規範

歸檔文件必須符合以下格式（可用 `~/Scripts/archive-session/normalize.py` 自動修正）：

1. 無行尾空白
2. `#` / `##` 標題行後保留一個空行
3. 每條原子筆記（`- **標題**：正文`）後保留一個空行
4. 無連續空行
5. 文件末尾恰好一個換行

歸檔追加新筆記時直接寫入正確格式，不依賴事後清洗。

```bash
# 清洗單個文件
python3 ~/Scripts/archive-session/normalize.py ~/Documents/Obsidian/archive/tech.md

# 清洗所有手動維護的歸檔文件
python3 ~/Scripts/archive-session/normalize.py ~/Documents/Obsidian/archive/{tech,mind,life,power,method,reading,projects}.md
```

注意：`index.md` 和 `cross-links.md` 為自動生成，不需手動清洗。`.idx` 文件為 hash 索引，由 `idx.py rebuild` 自動維護，不需手動編輯。
